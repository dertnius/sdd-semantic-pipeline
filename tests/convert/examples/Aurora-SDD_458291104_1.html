1. Introduction

1.1 Purpose

Aurora is an event-driven order fulfillment platform that orchestrates the lifecycle of customer orders from capture through delivery confirmation. It replaces the monolithic LegacyFulfillment system with a set of loosely-coupled services communicating over Apache Kafka. The primary goals are to reduce end-to-end fulfillment latency from ~45 minutes to under 5 minutes (p95), and to support a throughput of 12,000 orders/minute at peak.

This document describes the design at a level sufficient for implementation teams to build, test, and operate the system. It is intended for software engineers, SREs, QA engineers, and architecture reviewers.

1.2 Scope

In Scope
  • Order ingress and validation
  • Inventory reservation and release
  • Payment authorization orchestration
  • Warehouse dispatch and routing
  • Customer notification dispatch
  • Saga-based compensation flows
Out of Scope
  • Pricing & promotions engine (owned by PricingService)
  • Customer relationship management (CRM)
  • Returns & reverse logistics (Phase 2)
  • Tax calculation (delegated to Avalara)

1.3 Glossary & Definitions

Saga
A sequence of local transactions where each step publishes an event that triggers the next; failures trigger compensating transactions.
Outbox
A transactional outbox pattern used to atomically persist domain state and emit events.
Idempotency Key
A client-supplied unique key (Idempotency-Key header) that guarantees at-most-once side effects for retried requests.
SLO
Service Level Objective — a target value or range for a service level measured by an SLI.

2. Requirements

2.1 Functional Requirements

ID Requirement Priority Status
REQ-001 The system shall accept orders via a synchronous REST endpoint and acknowledge within 200ms. MUST DONE
REQ-002 The system shall reserve inventory before authorizing payment. MUST DONE
REQ-003 On payment failure, the system shall release reserved inventory within 30 seconds. MUST IN PROGRESS
REQ-004 The system shall deduplicate orders using a client-supplied Idempotency-Key. SHOULD DONE
REQ-005 The system should notify customers via email and SMS on each order state transition. COULD PLANNED

2.2 Non-Functional Requirements

ID Category Target
NFR-001 Throughput 12,000 orders/minute sustained, 18,000 burst
NFR-002 Latency End-to-end fulfillment < 5 min (p95), < 9 min (p99)
NFR-003 Availability 99.95% monthly (max ~21.9 min downtime)
NFR-004 Durability Zero order loss; RPO = 0, RTO ≤ 15 min
NFR-005 Security PCI-DSS SAQ-A scope; PII encrypted at rest (AES-256)

4. Architecture Overview

Aurora follows a choreographed saga pattern with a thin orchestration layer for compensation. Services are deployed as independent containers on Kubernetes (EKS) and communicate asynchronously through Kafka topics. Each service owns its datastore (database-per-service).

Design principle: No service reads another service's database directly. All cross-service state is exchanged via events or well-defined synchronous APIs.

The logical decomposition is as follows:

  1. Edge tier — API gateway, authentication, rate limiting.
    • Kong API Gateway
    • OAuth2 / OIDC via Keycloak
  2. Application tier — domain services.
    1. OrderIngressService
    2. FulfillmentOrchestrator
    3. InventoryService
    4. PaymentGateway (anti-corruption layer over Stripe)
    5. WarehouseService
    6. NotificationDispatcher
  3. Data tier — PostgreSQL clusters, Redis, Kafka, S3.

5.1 FulfillmentOrchestrator (detail)

The FulfillmentOrchestrator is the heart of the platform. It subscribes to order.received and drives the saga through the following states: RECEIVED → INVENTORY_RESERVED → PAYMENT_AUTHORIZED → DISPATCHED → COMPLETED. Any failure transitions to COMPENSATING and ultimately CANCELLED.

6. Data Model

Each service owns its schema. The core orders table in OrderIngressService's PostgreSQL instance is defined below.

orders schema (PostgreSQL 16)
CREATE TABLE orders (
    order_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    idempotency_key VARCHAR(64) NOT NULL,
    customer_id     UUID NOT NULL,
    status          order_status NOT NULL DEFAULT 'RECEIVED',
    total_cents     BIGINT NOT NULL CHECK (total_cents >= 0),
    currency        CHAR(3) NOT NULL DEFAULT 'USD',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_idempotency UNIQUE (customer_id, idempotency_key)
);

CREATE INDEX idx_orders_status ON orders (status)
    WHERE status IN ('RECEIVED', 'INVENTORY_RESERVED');

8. Runtime Flows

8.1 Happy-Path Order Fulfillment

The numbered steps below correspond to the sequence diagram:

  1. Customer submits an order to OrderIngressService.
  2. OrderIngressService persists the order and writes to its outbox table in the same transaction.
  3. A Debezium connector publishes order.received to Kafka.
  4. FulfillmentOrchestrator consumes the event and commands InventoryService to reserve stock.
  5. On reservation success, it commands PaymentGateway to authorize payment via Stripe.
  6. On authorization, it commands WarehouseService to dispatch.
  7. NotificationDispatcher sends a confirmation email and SMS.
Sequence (PlantUML)
@startuml
actor Customer
Customer -> OrderIngressService : POST /v1/orders
OrderIngressService -> Kafka : order.received
Kafka -> FulfillmentOrchestrator : order.received
FulfillmentOrchestrator -> InventoryService : ReserveStock
InventoryService --> FulfillmentOrchestrator : Reserved
FulfillmentOrchestrator -> PaymentGateway : Authorize
PaymentGateway -> Stripe : POST /charges
Stripe --> PaymentGateway : 200 OK
PaymentGateway --> FulfillmentOrchestrator : Authorized
FulfillmentOrchestrator -> WarehouseService : Dispatch
@enduml

8.2 Compensation (Payment Failure)

Critical path: If payment authorization fails after inventory is reserved, the orchestrator must emit inventory.release to avoid stranded reservations. See REQ-003 (target: release within 30s).

10. Architecture Decision Records

ADRs are recorded in MADR format and mirrored to the docs/adr/ folder in the aurora monorepo.

ADR-0001: Use Apache Kafka for inter-service messaging

Status ACCEPTED
Date 04 Mar 2025
Deciders Jane Smith, Raj Patel

Context. We need durable, ordered, high-throughput messaging between services with replay capability for the saga pattern.

Decision. We will adopt Apache Kafka (Amazon MSK) as the backbone, with topics partitioned by customerId to preserve per-customer ordering.

Consequences.

  • (plus) Replayability supports saga recovery and event sourcing.
  • (minus) Operational complexity; requires MSK expertise and partition-key discipline.

ADR-0002: Database-per-service over shared schema

Status ACCEPTED
Date 18 Mar 2025

Context. Shared databases create tight coupling and deployment contention across teams.

Decision. Each service owns a private PostgreSQL schema; cross-service data is exchanged only via events or APIs.

Consequences. Stronger autonomy and independent scaling, at the cost of eventual consistency and the need for the transactional outbox pattern.

ADR-0003: Synchronous payment authorization vs. fully async

Status PROPOSED
Date 22 Oct 2025
Supersedes —

Context. Stripe authorization typically completes in < 800ms, but tail latencies can reach 6s. We must decide whether the orchestrator blocks or treats authorization asynchronously.

Decision. Pending review by Marcus Osei.

11. Quality Attributes (NFRs)

Performance

SLI: p95 fulfillment latency. SLO: < 5 min. Measured via OpenTelemetry traces exported to Honeycomb.

Security

Card data never touches Aurora; tokenized at the edge via Stripe Elements. PCI-DSS SAQ-A scope.

Observability

Structured JSON logs to Loki; metrics to Prometheus; distributed tracing via OTel. Each event carries a traceparent.

“The system is designed for graceful degradation: if NotificationDispatcher is down, orders still complete — notifications are queued and retried with exponential backoff.” — Jane Smith, design review 30 Sep 2025

13. Risks, Assumptions & Open Questions

13.1 Risks

RISK-01 (High): Kafka partition skew on hot customers could cause consumer lag. Mitigation: composite partition key (customerId, orderId) under evaluation.

13.2 Open Questions / Action Items

  • Confirm MSK broker count for prod (decided: 6 brokers).
  • Marcus Osei to finalize ADR-0003 (sync vs async payment) by 28 Nov 2025.
  • Li Chen to validate RTO ≤ 15 min via game-day chaos test.

13.3 Assumptions

  1. Stripe SLA of 99.99% is sufficient for the payment critical path.
  2. Average order contains ≤ 25 line items.
  3. The 3PL provider can sustain 12k dispatch requests/minute.
